# AGENTS.md

## 目的

- このファイルは、`visual-pitch-trainer` リポジトリで作業するコーディングエージェント向けのリポジトリ固有ルールを定義する。
- 変更はできるだけ具体的かつ最小限にし、現在の静的 Web アプリ構成を崩さないことを優先する。
- 実装に入る前に、関連する `README.md`、`docs/design.md`、`docs/implementation-notes.md` を確認すること。

## リポジトリ概要

- 難聴者向けに、音程練習を視覚的に支援する Web アプリ。
- Bun + Vite + TypeScript で開発し、ビルド後は `dist/` だけで動く静的 Web アプリとして配信する。
- サーバは持たず、参考音再生、マイク入力、音程検出、グラフ描画、練習サマリーはブラウザ内で処理する。
- 主要ファイル:
  - `index.html`: アプリの HTML。
  - `styles.css`: 画面全体のスタイル。
  - `src/app.ts`: UI、Web Audio API、マイク制御、描画の統合。
  - `src/pitchMath.ts`: 音名、MIDI、周波数、音程差などの計算。
  - `src/pitchDetection.ts`: `pitchy` を使った音程検出。
  - `src/graphModel.ts`: グラフ描画用の座標・表示範囲計算。
  - `src/session.ts`: 練習サンプルとサマリー計算。
  - `docs/`: 設計メモと実装メモ。

## 開発環境前提

- 日常的な開発は Dev Container 上で行う前提とする。
- パッケージ管理とスクリプト実行は Bun に統一する。
- Bun の期待バージョンは `.bun-version` と `package.json` の `packageManager` を参照する。
- ローカル起動は `bun run dev` を使う。Vite はコンテナ内の 5173 番で起動し、Dev Container ではホスト側の `http://localhost:35173/` で確認する。
- マイク入力はブラウザや表示方法に依存するため、挙動確認は可能な限り外部ブラウザで行う。

## 仕様の優先順位

- プロダクト目的と体験設計: `README.md`、`docs/design.md`
- 現在の実装方針と既知の注意点: `docs/implementation-notes.md`
- 実際のスクリプトと依存関係: `package.json`
- 実装とドキュメントが食い違う場合:
  - バグ修正では、まず現行の実ランタイム挙動を確認して解釈する。
  - 新機能や仕様拡張では、原則として `docs/design.md` の方針を優先する。
  - 仕様、セットアップ、音声処理方針に影響する変更をした場合は、関連ドキュメントも更新する。

## 作業方針

- 変更前に、影響を受けるファイルを確認してから手を入れる。
- 依頼に直接関係しない大規模リファクタや依存追加は避ける。
- 既存の命名、構成、責務分割を尊重する。
- 症状への場当たり対応より、原因に対する修正を優先する。
- ユーザーへの応答、コミットメッセージ、Pull Request のタイトルと本文は、特段の指定がない限り日本語で記述する。
- GitHub 上の確認や操作が必要な場合は、原則として `gh` コマンド等の CLI を使用する。
- `dist/` 配下の生成物は、ユーザーから明示的に求められない限り手編集しない。

## このプロジェクト固有の制約

### 静的アプリ構成

- 初期バージョンはサーバなしの静的 Web アプリとして維持する。
- 明示的な依頼なしに API サーバ、DB、外部状態ストア、認証基盤を導入しない。
- 音声データをサーバへ送信する設計にしない。
- 履歴保存を追加する場合は、まず `localStorage` または `IndexedDB` などブラウザ内保存を検討する。

### 音声処理

- 参考音再生は Web Audio API の `OscillatorNode` を前提とする。
- マイク入力は `getUserMedia` と `AnalyserNode.getFloatTimeDomainData()` を前提とする。
- 音程検出は現在 `pitchy` の McLeod Pitch Method を使っている。強い理由なしに手書きアルゴリズムへ戻さない。
- 音量が小さい、検出不能、参考音の回り込み、オクターブ誤認などの状態を UI とロジックの両方で丁寧に扱う。
- マイク権限、HTTPS/localhost 制約、ブラウザ差異を考慮する。

### UI とアクセシビリティ

- 難聴者向けの視覚支援アプリであるため、音だけに依存したフィードバックにしない。
- 状態表現は色だけに依存せず、位置、形、数値、文言を組み合わせる。
- PC、タブレット、スマートフォンで読みやすく操作しやすいレスポンシブ UI を保つ。
- 発声中に見る画面なので、主要情報は視認性を優先する。
- UI を変更した場合は、狭い画面幅でも文字や操作部品が重ならないことを確認する。

## 実装ガイド

- `src/app.ts` は UI と Web Audio API の統合点として扱い、肥大化する変更では責務を既存モジュールへ切り出せないか検討する。
- 音名、周波数、MIDI、cents 差分の計算は `src/pitchMath.ts` に寄せる。
- 音程検出ライブラリの呼び出しや候補選択は `src/pitchDetection.ts` に寄せる。
- グラフの表示範囲、座標変換、サンプル抽出は `src/graphModel.ts` に寄せる。
- 練習サンプルやサマリーの集計は `src/session.ts` に寄せる。
- DOM 構造を変える場合は `index.html`、`styles.css`、`src/app.ts` の対応を必ずそろえる。
- テストしやすい計算ロジックは DOM や Web Audio API から分離する。

## 検証

コードを変更した場合は、変更範囲に応じて以下を実行する。

- lint:
  - `bun run lint`
- unit test:
  - `bun run test`
- typecheck:
  - `bun run typecheck`
- production build:
  - `bun run build`

推奨の基本順序:

```sh
bun run lint
bun run test
bun run typecheck
bun run build
```

- UI や音声まわりを変更した場合は、可能なら `bun run dev` で起動し、ブラウザで実操作を確認する。
- マイク入力や実機ブラウザでしか確認できない挙動を検証できなかった場合は、最終報告で未検証事項として明示する。
- `tmp/`、`dist/`、`node_modules/` は通常の lint 対象外または生成物として扱う。

## エージェント向け安全策

- テキスト検索は `rg` を優先する。
- ファイル編集は `apply_patch` を優先する。
- 破壊的な git 操作は避ける。
- ワークツリーが dirty でも、既存変更を巻き戻さず共存させる。
- 依頼範囲外のユーザー変更を上書きしない。
- 判断に迷う場合は、安全な仮定で進められる範囲を進め、その仮定を最終報告で明示する。
