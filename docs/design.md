# 設計

## 概要

Visual Pitch Trainer は、難聴者向けに音程練習を視覚的に支援するWebアプリである。

利用者は参考音（ド・レ・ミなど）を再生し、その音に合わせて発声する。アプリはマイク入力をリアルタイムに解析し、目標音との差や時間経過による音程の上下を画面上に表示する。

音を聞いて判断することを前提にせず、視覚的なフィードバックを中心に設計する。

表示フォントには Google Fonts の利用を許容する。また、ボーカル抽出や RMVPE / CREPE 推定を有効にした場合は、ブラウザ内で使う ONNX Runtime Web の WASM ファイルやモデルを取得するための外部通信を許容する。音源解析用のピッチ推定モデルは公開成果物に含める。音声データ、マイク入力、解析結果を外部へ送信する設計とは切り分けて扱う。

## 主な要件

- 難聴者向けの音程練習ができること
- 参考音を再生しながら発声練習できること
- 発声した音程をリアルタイムに分析できること
- 目標音とのズレを視覚的に把握できること
- 時間経過による音程の上下が見えること
- PC、タブレット、スマートフォンで動作すること
- サーバなし、静的ホスティングのみで動作すること

## 体験設計

中心となる体験は、以下の流れとする。

1. 利用者が目標音を選ぶ
2. アプリが参考音を再生する
3. 利用者が参考音に合わせて発声する
4. アプリがマイク入力から音程を検出する
5. 目標音との差や高低の傾向を表示する
6. 発声中の音程変化をリアルタイムグラフで表示する

この体験により、利用者は「今の音が高いか低いか」だけでなく、「発声中に音程が上がっているか、下がっているか、揺れているか、安定しているか」を視覚的に確認できる。

## 主要画面

### 練習画面

現在の中心画面。

表示する情報:

- 目標音
- 音域選択（C3-C4: 低めの男性声 / C4-C5: 高めの男性声・低めの女性声 / C5-C6: 高めの女性声）
- 参考音再生ボタン
- 参考音量
- マイク開始・停止
- 目標音との音程差
- 高い・低い・合っている状態
- リアルタイムピッチグラフ

操作:

- 声の高さの目安つきで C3-C4 / C4-C5 / C5-C6 の音域を選ぶ
- 半音を含む1オクターブ分の目標音から選ぶ
- 参考音を再生する
- 参考音量を調整する
- マイク入力を開始する
- 練習を開始・終了する

### 設定項目

現在の画面では、練習に必要な設定を同じ画面内で扱う。

- 判定の幅: 半音の何%を許容するか。難聴者・初心者向けの初期値は緩めにする。
- マイク感度
- 音域: C3-C4 / C4-C5 / C5-C6 など。初心者向けに「低めの男性声」などの目安を併記する。
- 参考音量

今後の拡張候補:

- 基準ピッチ A4 = 440Hz / 442Hz など
- 音名表記: ドレミ / CDE
- 色覚特性に配慮したテーマ
- 参考音の音色

### 履歴

現在は練習サマリーを画面上に表示する。履歴保存を追加する場合は、以下を扱う。

- 練習日時
- 練習した目標音
- 平均の音程差
- 安定して発声できた時間
- 音程グラフの簡易振り返り

### 音源の声域推定

ローカルの音源ファイルまたは動画ファイルを読み込み、ブラウザ内で歌声らしい音程の範囲を推定表示する。

伴奏入りの通常楽曲を直接解析すると、ベース、ギター、シンセ、ハモリなどの音程を拾って声域推定やグラフが乱れやすい。そのため、WebGPU 対応ブラウザでは Demucs 系モデルや RoFormer 系モデルによるボーカル抽出を任意で行い、抽出後のボーカル音声を声域推定に渡す。さらに、WebGPU 対応ブラウザでは CREPE や RMVPE の ONNX モデルによるニューラルピッチ推定を任意で比較できるようにする。WebGPU 非対応環境や処理が重すぎる端末では、抽出済みボーカル音源を読み込ませるか、従来の軽量なピッチ検出を使う運用も許容する。

表示する情報:

- 推定声域
- よく出る範囲
- 中心付近の音
- 検出できた歌声らしい区間の割合
- 時間経過に沿った推定ピッチの点群と中心線
- 音源再生中の現在位置を示す縦線

将来的な拡張候補:

- Web Worker によるバックグラウンド解析
- ボーカル抽出モデルのローカルキャッシュ
- 解析結果のローカル保存

## 視覚フィードバック

難聴者向けであるため、音の状態を色だけに依存して表現しない。色、位置、形、数値、文言を組み合わせる。

Canvas のグラフは主要な視覚表現だが、状態文、数値、サマリー、必要に応じた補助テキストで内容を補う。

### 現在音程メーター

現在は採用せず、リアルタイムピッチグラフを主役にしている。将来的に復活させる場合は、目標音を中央に置き、現在の音程が高ければ上、低ければ下に表示する。

- 中央: 目標音
- 上方向: 目標より高い
- 下方向: 目標より低い
- 中央付近: 許容範囲内

### リアルタイムピッチグラフ

横軸を時間、縦軸を音程とする。

- 目標音を水平線で表示
- 発声した音程を線で描画
- 無音や検出不能区間は線を途切れさせる
- 許容範囲を帯で表示する
- 目標音に近づくほど目標ラインを発光させる
- 許容範囲に入った瞬間は波紋で着地を示す

このグラフにより、音程の上ずり、下がり、揺れ、安定度が見える。

### 状態表示

状態は短い文言と形で示す。

- 低い
- 合っている
- 高い
- 音が小さい
- 検出できない
- 連続して合っている時間

## 音声処理

ブラウザ内で Web Audio API を使う。

### 参考音再生

現在は OscillatorNode による正弦波で実装している。

参考音量は画面上のスライダーで調整できる。スピーカーからの回り込みを抑えたい場合にも使う。

今後の拡張候補:

- ピアノ音源
- 声に近いガイド音
- 楽器音の切り替え
- 短音再生
- 継続再生

### マイク入力

`getUserMedia` でマイク入力を取得する。

利用時の注意:

- HTTPS環境が必要
- ローカル開発では localhost で利用可能
- ユーザー許可が必要
- スマートフォンではブラウザ差異を確認する必要がある

### 音程検出

検討したアルゴリズム:

- YIN
- autocorrelation
- McLeod Pitch Method

手書き実装ではなく、ブラウザ対応の既存ライブラリを使う。マイクのリアルタイム音程検出は、ライセンス面とブラウザでの扱いやすさを考慮し、`pitchy` の McLeod Pitch Method を使う。音源ファイル解析では、軽量な `pitchy` に加えて、WebGPU 対応環境向けに RMVPE / CREPE の ONNX モデルを選べるようにする。精度、遅延、スマートフォンでの処理負荷を見ながら調整する。

### 音程変換

検出した周波数 Hz を以下に変換する。

- MIDI note number
- 音名
- 目標音との音程差

内部計算では cents 差分を使う。画面上では初心者にも分かりやすいように「半音の何%くらいの差か」として表示する。

### 音源ファイル解析

音源ファイル解析は `File API` と `AudioContext.decodeAudioData()` でローカルファイルを読み込む。動画ファイルやブラウザ標準で読み込めない形式では、Mediabunny を使って音声トラックを demux/decode し、PCM データへ変換する。得られた音声はモノラルにミックスしてから短いフレーム単位でピッチ検出する。

ボーカル抽出を有効にした場合は、Demucs 系モデル、RoFormer 系モデルを ONNX Runtime Web / WebGPU で実行し、抽出されたボーカル音声を声域推定へ渡す。RoFormer 系は vocal / instrumental の2 stem特化モデルとして扱い、品質検証用に BS-RoFormer を選べるようにする。音源ファイル解析のピッチ推定では、既定は RMVPE としつつ、通常のPCやスマートフォンなど処理負荷が気になる環境では pitchy、比較検討では CREPE small / medium / large / full も選べるようにする。RMVPE / CREPE は ONNX Runtime Web / WebGPU で実行し、歌声向けのピッチ候補を使う。音声データはサーバへ送信しない。

## データ設計

現在は練習サマリーを画面上に表示する。履歴機能を追加する場合は、練習データをローカル保存する。

保存先候補:

- localStorage
- IndexedDB

リアルタイム音声そのものは保存しない。必要に応じて、音程サンプルや集計値のみを保存する。

```ts
type PracticeSession = {
  id: string;
  targetNote: string;
  targetFrequency: number;
  startedAt: string;
  durationSec: number;
  averageErrorCents: number;
  stableDurationSec: number;
  accuracyRate: number;
  pitchSamples: PitchSample[];
};

type PitchSample = {
  timeMs: number;
  frequency: number | null;
  note: string | null;
  centsFromTarget: number | null;
  volume: number;
};
```

## サーバ構成

### 現在の構成

サーバなしの静的Webアプリとして構成する。

```text
Static Hosting
  └─ Browser App
      ├─ Reference Tone Playback
      ├─ Microphone Input
      ├─ Pitch Detection
      ├─ Visualization
      └─ Practice Summary
```

この構成の利点:

- 実装・運用がシンプル
- デプロイが簡単
- 低遅延
- 音声データをサーバに送らない
- 運用コストが低い
- PC、タブレット、スマートフォンで利用しやすい

想定配信先:

- GitHub Pages
- Cloudflare Pages

### 今後の拡張候補

アカウントや同期が必要になった段階で、APIサーバとデータベースを追加する可能性がある。

```text
Browser
  ├─ Realtime Pitch Analysis
  ├─ Visualization
  └─ Practice UI

API Server
  ├─ User Account
  ├─ Practice History
  ├─ Lesson Content
  └─ Progress Analytics

Database
  ├─ Users
  ├─ Sessions
  ├─ Pitch Samples Summary
  └─ Training Programs
```

リアルタイム音程解析は、将来も基本的にはブラウザ側に置く。サーバへ音声を送り続ける構成は、遅延、通信量、プライバシー、実装コストの面で重いため、現在の構成には含めない。

サーバで扱う可能性があるもの:

- アカウント管理
- 練習履歴の同期
- 複数端末対応
- トレーニングメニュー配信
- 先生・支援者との共有
- 長期的な成長分析
- 有料プラン管理

## 設計上の注意

- 音だけに依存しない
- 色だけに依存しない
- スマートフォンでも読みやすい文字サイズにする
- 発声中に見やすいよう、主要情報を大きく表示する
- 無音、音が小さい、検出不能の状態を明確に出す
- マイク許可が必要な理由を画面内で簡潔に伝える
- 音声データをサーバ送信しない方針を明示する
- リアルタイム処理は遅延を小さくする
- 練習後に自分の音程変化を振り返れるようにする
