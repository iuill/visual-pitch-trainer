# 実装メモ

## 現在の実装方針

現在は Bun + Vite + TypeScript を使い、ビルド後の `dist/` を静的ホスティングへ配信する構成にしている。

構成:

```text
index.html
styles.css
src/app.ts
vite.config.ts
tsconfig.json
docs/
```

ローカル確認は Dev Container 内の Bun + Vite を使い、可能であれば外部ブラウザで開く。

Dev Container では以下で起動できる。

```sh
bun run dev
```

このコマンドは Vite の開発サーバを起動し、通常は `http://localhost:5173/` でアプリを配信する。

確認用コマンド:

```sh
bun run typecheck
bun run build
```

`bun run build` で生成される `dist/` を GitHub Pages または Cloudflare Pages に配信する。

## 実装済みの主な機能

- 目標音の選択
- 音域の選択（C3-C4 / C4-C5 / C5-C6、声の高さの目安つき）
- 半音を含む1オクターブ分の目標音選択
- 参考音の継続再生と停止
- 参考音量の調整
- ドレミ再生
- マイク入力の開始・停止
- 複数マイクがある場合の入力デバイス選択
- マイク音量と検出信頼度の表示
- 音程の時間変化グラフ
- 目標音を中心に上下1オクターブの音名レーンを表示
- 練習サマリーの表示

## UI方針の変更

当初は「現在の音程」パネルを大きく表示していたが、リアルタイム音程は声の揺れ、倍音、ノイズ、マイク入力の影響で激しく変化しやすい。

そのため、瞬間的な音程表示は利用者にとって分かりにくいと判断し、現在は削除している。

現在の主役は「音程の時間変化」グラフである。

残している補助情報:

- 音量
- 信頼度
- 状態メッセージ
- 練習サマリー

## マイク入力について

ブラウザの制約により、マイク入力は表示方法に依存する。

確認方針:

- `file://` で直接開くと動くブラウザもあるが、安定しない
- Vite が提供する `http://localhost:5173/` などのURLを外部ブラウザで開くのが望ましい
- GitHub Pages や Cloudflare Pages のような HTTPS 静的配信ではマイク利用が可能

複数マイクがある場合に、どのマイクを使っているか分かりにくかったため、入力マイク選択UIを追加した。

注意:

- マイク権限を許可する前は、ブラウザ仕様によりデバイス名が表示されないことがある
- 一度マイク開始を許可すると、実デバイス名が表示されることが多い

## 参考音について

参考音は Web Audio API の `OscillatorNode` で生成している。

現在の動作:

- `参考音` ボタンを押すと、現在の目標音を継続再生する
- `停止` ボタンを押すと再生を止める
- 再生中に目標音を変更すると、鳴っている参考音も新しい目標音に追従する
- 参考音量スライダーで継続再生中の音量も変更できる
- `ドレミ再生` は短音を順番に鳴らす

注意:

スピーカーから参考音を鳴らしながらマイク解析すると、マイクが参考音を拾い、発声音と混ざる。音程検出の安定性に影響するため、将来的には以下を検討する。

- ヘッドホン利用を案内する
- 参考音再生中の解析を補正する
- 参考音と発声練習を時間的に分ける

## 音程検出の経緯

初期実装では、自己相関に近い簡易的な音程検出を実装した。

その後、以下の問題があった。

- 音程が激しく変化する
- 倍音を基音として誤認する
- オクターブ違いが起こる
- 参考音の回り込みに弱い
- C4の参考音と、発声がC4ラインに乗る感覚にズレがある

改善として、手書きのYINアルゴリズムに差し替えた。

さらに、目標音が既知であることを利用し、YINの候補選択で以下を行った。

- 目標音の上下1オクターブに探索範囲を絞る
- 複数の周期候補から、信頼度と目標音からの近さを見て選ぶ

ただし、手書き実装の調整には限界があったため、現在は `pitchfinder` の YIN と McLeod Pitch Method を併用している。どちらもブラウザで `Float32Array` を直接扱えるため、Web Audio API のマイク入力と相性がよい。

現在の処理:

- `AnalyserNode.getFloatTimeDomainData()` で取得した波形を `pitchfinder` に渡す
- YIN と McLeod の候補を取得する
- 目標音の上下1オクターブに候補を絞る
- 信頼度と目標音からの近さを使って候補を選ぶ
- 音量が小さい場合は検出しない

## 技術構成

素のHTML/CSS/JavaScriptから、Vite + TypeScript 構成へ移行済み。

Node.js は開発・ビルドに使うが、サーバアプリにするわけではない。

```text
開発時:
Node.js / Bun + Vite + TypeScript + npm package

ビルド後:
HTML / CSS / JavaScript の静的ファイル

配信:
GitHub Pages または Cloudflare Pages
```

GitHub Pages や Cloudflare Pages への静的配信は、この方針と相性がよい。

## 音程検出ライブラリ

現在の採用:

- `pitchfinder`

候補として残しているもの:

- `pitchy`
- WASMベースの pitch detection library

検討順:

1. `pitchfinder` の YIN と McLeod を実測する
2. 安定しなければ `pitchy` の McLeod Pitch Method 系を試す
3. さらに必要であれば WASM 系を検討する

選定基準:

- ブラウザでリアルタイム動作できること
- マイク入力の `Float32Array` を直接扱えること
- 音声・歌声・発声に対して安定していること
- オクターブ誤認が少ないこと
- Viteでバンドルできること
- GitHub Pages / Cloudflare Pages の静的配信で動くこと

## 静的配信方針

理想の配信先:

- GitHub Pages
- Cloudflare Pages

Vite構成では、`bun run build` で生成される `dist/` を配信すればよい。

サーバ側で音声解析を行う予定はない。リアルタイム音程検出はブラウザ内で行う。

理由:

- 低遅延
- プライバシー面で扱いやすい
- 音声データをサーバへ送らない
- 静的ホスティングで運用できる
- コストが低い
